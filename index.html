The Reframe: Agent IS the Analyst

"We're not building software for public markets analysts. We are building a public markets analyst. An analyst is an information processing and presentation machine — your job is to respond to requests for information and suggest good ideas based on what we're observing in the market. Come pitch me an idea as a PM."

V1 IS

  • An analyst agent — does the analytical work itself
  • Processes information requests from PMs
  • Generates investment recommendations
  • Pitches investment ideas based on observed market signals
  • Multi-channel: Google Chat, custom UI, more
  • Chrome extension for one-click ingestion from any web page
  • Grounded in Vertex AI Search over indexed documents

V1 is NOT

  • NOT a tool that helps an analyst — the agent is the analyst
  • NOT a workflow management UI with shiny dashboards
  • NOT a complex spiderweb visualization
  • NOT quantitative — Bloomberg/Reuters feeds out of scope
  • NOT portfolio construction or attribution
  • NOT AI personas or morning briefings
  • NOT an AI-generated sell-side report (that's "beta", no edge)

Atlas Live MCP Queries — Fresh Data, No Duplication

Atlas does not just rely on what's been ingested. At query time, the agent reaches out to Fireflies and Google Workspace through their MCP servers and pulls fresh data live. Meeting transcripts and PM chat conversations stay in their source systems — Atlas queries them on demand.

🎙 Fireflies MCP — Live Meeting Queries

Atlas hits the Fireflies MCP server at api.fireflies.ai/mcp to retrieve transcripts, summaries, and action items in real time.

Example PM queries Atlas can answer live:

  • "What did Sean say about Discovery in yesterday's call?"
  • "Summarize all the meetings we had with Standard Bank management last quarter."
  • "Did anyone raise concerns about FirstRand's NPL ratio in the last 30 days?"
  • "What action items came out of the Tuesday strategy session?"

Tools used by Atlas:

fireflies_search fireflies_get_transcripts fireflies_get_transcript fireflies_get_summary fireflies_fetch

💬 Google Workspace MCP — Live Chat Queries

Atlas queries Google Chat (and Gmail, Drive, Calendar) live via the Google Workspace MCP. The PM chatter stays where it is — Atlas reaches into it on demand.

Example PM queries Atlas can answer live:

  • "What has the team discussed about FirstRand in the chat this week?"
  • "Find the chart Sean shared about ZAR volatility two weeks ago."
  • "What sell-side reports have we received about Discovery this month?"
  • "Pull up the email thread where Rob flagged the SA macro concern."

Tools used by Atlas:

chat.search_messages chat.list_spaces gmail.search_threads drive.search_files calendar.list_events

Why live queries instead of pre-ingestion?

Always fresh. Meeting from this morning is queryable immediately. No nightly sync delay. No "is this stale?" question.
No duplication. Fireflies already stores transcripts. Google already stores chat. Atlas reads from the source of truth — no parallel database to maintain.
Permission-aware. MCP queries flow through OAuth/API key auth — Atlas only ever sees what the authenticated user is allowed to see.

The Rezco Chrome Extension — Primary Ingestion Path

Inspired by Otter.ai's side-panel pattern. Opens on any web page. Reads the DOM directly. PMs clip what they read — no behavior change, no integrations with sell-side portals required.

How It Works

  1. PM is browsing Bloomberg, Gmail, an IR website, or reading a PDF
  2. Clicks the Rezco extension icon → side panel slides in from the right
  3. Extension detects content type via DOM inspection: email, webpage, PDF
  4. First-pass enrichment: extracts companies, themes, industries from the DOM
  5. PM confirms tags (or just clicks "Ingest")
  6. Payload POST'd to Rezco's Go ingestion microservice
  7. Microservice normalizes (text or document), pushes to Vertex AI Search, files in Drive folder, links entities
  8. Atlas can immediately query the new content via grounded RAG

Why DOM-Based?

  • Quickest path to ingest — pulls structured content directly from the page
  • Precise content type detection — JavaScript runs against DOM to differentiate email vs. webpage vs. PDF
  • No portal integrations needed — bypasses Morgan Stanley sell-side portal complexity
  • Internal-only — no Chrome Web Store approval
  • Behavior-preserving — PMs already read these pages, this just adds a click
  • First-pass enrichment — happens client-side before payload is sent
Inspiration: Otter.ai side-panel pattern (opens on any page) + ID8 extension being built by Ruan (DOM-based, ATIO-loop inspired)

Extension → Microservice Data Flow

DOM Read → Type Detection → Tag Extraction → POST /ingest → Go Microservice → Vertex AI Search

Go Ingestion Microservice

A single-purpose Go service that receives DOM payloads from the Chrome extension, normalizes them, and routes them to Vertex AI Search and Google Drive. Stateless, horizontally scalable.

Endpoint Purpose Behavior
POST /ingest/text Receives extracted text from DOM (web pages, emails) Tags entities, embeds, indexes in Vertex AI Search, returns ingestion ID
POST /ingest/document Receives full PDF/document binary Parses, extracts text + metadata, files in Drive folder by asset, indexes in Vertex AI
POST /ingest/clip Receives a partial selection from a page Stores as a quote/note linked to the source URL, indexes for later retrieval
GET /assets/{figi}/documents Retrieve all documents for an asset Returns indexed documents tagged to the FIGI
POST /tag/suggest Get tag suggestions for a payload Returns company / theme / sector candidates from a vector similarity search

Vertex AI Search — The Indexed Knowledge Base

All ingested content (clipped pages, PDFs, transcripts, chat messages) flows into Vertex AI Search where it is indexed and made queryable via grounded RAG. Atlas accesses Vertex AI Search to provide smart, source-cited answers.

What gets indexed

  • PDF documents (sell-side, IR packs, annual reports)
  • Web page text clipped via the extension
  • Email content (sell-side reports from Gmail)
  • Meeting transcripts (Fireflies)
  • Google Chat messages (PM discussions)
  • Internal Rezco notes added directly

How Atlas queries it

  • Grounded RAG — every answer cites sources
  • Cross-source retrieval (PDFs + transcripts + chat)
  • Filter by FIGI / asset / theme / sector
  • Temporal filtering (most recent first by default)
  • Confidence scoring on retrieved chunks
  • Conflict detection across sources

Drive → Vertex Sync Agent

A small, single-purpose sub-agent that watches a designated Google Drive folder for new uploads. When a document lands in Drive, the agent picks it up and pushes it into Vertex AI Search for indexing — making it immediately queryable by Atlas. Rezco doesn't currently have a Drive set up; V1 will provision the watched folder and the sync agent will populate the index from there onward.

What it does

  • Watches a designated Google Drive folder for new files
  • Detects new uploads via Drive change notifications / poll
  • Downloads the file (PDF, Doc, Sheet, image) and parses text
  • Extracts entities (companies, sectors, themes) for tagging
  • Pushes the parsed content into Vertex AI Search
  • Links indexed records back to the original Drive file
  • Logs every sync for audit and re-indexing later

Why this is the right shape

  • Single responsibility — it only does one thing: Drive → Vertex AI Search
  • Behavior-preserving — anyone (PM or junior staff) can drop a file in Drive and it just becomes queryable
  • No complex filing logic — Rezco can organize the Drive however they want; the agent indexes everything in scope regardless of subfolder
  • Idempotent — re-running on the same file is safe; it updates the existing index entry
  • Complements the Chrome Extension — extension handles "I'm reading this now"; Drive sync handles "this got emailed to us / saved manually"

Strategic Context

Rezco Asset Management (est. 2005, Johannesburg) is executing a radical shift: replacing the entire human analyst layer with an AI-powered research agent. The AI serves as an analyst, not a portfolio manager — it provides decision support to PMs Rob, Simon, and Sean.

The Problem

Portfolio Manager Bottleneck

  • PMs spend excessive time training and motivating junior analysts
  • High analyst churn rate wastes training investment
  • More human analysts = more management overhead, not more alpha
  • Human analysts are "bad filters" — strip out signals they perceive as noise

Information Fragmentation

  • Data scattered: Google Chat, Bloomberg, emails, meetings, calendars
  • No centralized logging of investment debates and decisions
  • Ideas stagnate in archives — incomplete work goes untracked
  • Geopolitical noise distracts from specific investment signals

The Solution: AI Analyst Agent

An orchestrator-based system of specialized AI sub-agents that ingest, map, analyze, and surface investment intelligence — grounded in Rezco's own philosophy.

What it does: Ingests all data sources, builds a non-linear knowledge graph ("Spiderweb"), provides prompted and unprompted analysis through multiple AI personas, tracks conviction over time, detects contradictions, and surfaces high-value signals.
What it does NOT do: Make buy/sell decisions. Generate alpha recommendations. Act as a portfolio manager. Operate as a black box. The PM remains the decision-maker.

Key Architectural Principles

  • Architecture at inception matters — biggest cost is PM time training the AI
  • Flexible foundation — add capabilities without rebuilding
  • Not a black box — sources cited, reasoning transparent
  • Reduce PM friction — every feature saves time, doesn't add it
  • Continuous, not episodic — state maintained across months/years
  • Proprietary vs. public always distinguishable — compliance requirement
  • Analyst, not PM — information and analysis only
  • Buy-side is different — always relative, cross-sector, never siloed
  • Benchmark-agnostic, not benchmark-ignorant
  • Signal over noise — amplify faint signals, filter the rest
  • Ideas must not die in the archive — active monitoring
  • Cognitive diversity by design — multiple AI personas

Headwinds/Tailwinds Framework

Core Rezco investment philosophy that the AI must deeply understand and apply.

Tailwind Environment

Positive factors outweigh negatives. Companies underpromise and overdeliver. Environment is "good and getting better." Management provides positive news incrementally.

Headwind Environment

Binary and high-risk. Management drip-feeds bad news. "It's tough out there," "consumers are struggling." Hedging bets. Hoping conditions improve.

Capability Description
Longitudinal Earnings Call Analysis Detect shifts in management tone over time — from bullish to defensive or vice versa
Linguistic Cue Detection Track specific phrases: "it's tough out there," "consumers are struggling," "we continue to see improvement"
H/T Classifier Running classification per company: net headwind, net tailwind, or transitioning
Inflection Point Detection Find points where long-term projects "bear fruit" — not yet in financial models

Contradiction & Temporal Intelligence

Conviction Tracking

  • Record every PM view: timestamp, instrument, direction, confidence
  • Build conviction curve per instrument per PM
  • Surface when conviction changed without acknowledgment

Contradiction Detection

  • Current views vs. historical views on same instrument
  • Views vs. portfolio positioning ("worried about SA macro but overweight retailers")
  • Management statements vs. financial results
  • Stated intentions vs. actual execution

Build Roadmap

Pre-Sprint: Discovery & Access (1 Week)

Simon walks Techbridge through the research hub. Identify what's used vs. unused. Get access to both consoles. Inventory APIs, pipelines, Vertex AI/Spanner infrastructure. Map data source hierarchy to trust layer.

Sprint 1-2: Foundation (Integrate with Existing)

Connect orchestrator to existing instrument DB, Vertex AI doc store, Spanner vector DB. Knowledge Graph schema with non-linear linking + temporal indexing. Extend RAG with graph traversal. Idea Registry service.

Sprint 3-4: Core Analyst + Automation

Research & Retrieval Agent. Google Chat Listener. Meeting Transcription Pipeline. Entity resolution (FIGI/ISIN). Classification Automation Agent. Custom Field Maintenance Agent. Basic signal scoring.

Sprint 5-6: Intelligence Layer

Temporal indexing + conviction tracking. Contradiction detection. Source weighting (extend existing hierarchy). Comparison Agent (GARP). Headwind/tailwind classification engine.

Sprint 7-8: Proactive Capabilities

Morning Briefing Agent. Holdings Sync. Financial Data Store (PE, prices, revisions). Threshold alerts. Execution Gap Detection. Benchmark deviation tracking.

Sprint 9-10: Conviction & Signal Layer

Ideas → Investments pipeline. Signal amplification + manual override. Cognitive bias detection. Longitudinal sentiment on earnings calls. Linguistic cue detection. Research curation agent.

Sprint 11-12: Advanced Features

7 AI Personas (Buffett, Dalio, Druckenmiller, Gundlach, Robin Hood, Technical). Report Generation (H/T + sentiment). Meeting Prep. Model Rolling. Screening (H/T + benchmark gaps). Buy/sell automation.

Sprint 13-14: Front-End Expansion

Morning briefing dashboard. Instrument deep-dive (conviction curves, H/T). Ideas pipeline (kanban). Portfolio overview. Signal feed. Knowledge graph explorer. Persona comparison. Console modernization.

Global allocation (macro personas). 24/7 market screening. Real-time synchronous debate participation. Performance attribution (Phase 3). Risk factor analysis (Phase 3). Analyst reduction as agents mature.
